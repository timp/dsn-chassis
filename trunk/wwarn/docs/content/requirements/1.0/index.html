1. Introduction

1.1. Scope of Chassis/WWARN

1.1.1. WWARN Data Management

The Chassis/WWARN system is intended to support the data management activities of the WWARN data-sharing network. These data management activities include the submission of data, curation of submitted data to meet quality control and other (e.g., ethical) criteria as defined by WWARN, and mapping of submitted data onto WWARN standard data dictionaries.

The end goal of these data management activities is to produce data that is suitable for visualisation, analysis and publication by The WWARN Application.

1.1.2. Chassis/WWARN and The WWARN Application

Note that the Chassis/WWARN system is distinct from The WWARN Application (current prototype at https://www.wwarn.org/warnapp/datamap/datamap.html). The WWARN Application is a separate system, intended to support the visualisation, analysis, aggregation, sharing and publication of standardised data.

This specification assumes that data will be periodically exported from the Chassis/WWARN system and ingested into The WWARN Application. Management of this export and ingest process is out of scope for the Chassis/WWARN system, and will be specified elsewhere. Requirements for The WWARN Application, including all requirements for data visualisation, aggregation and analysis tools, are not in scope for this document, and are specified elsewhere.

1.1.3. First Major Release (1.0)

This document specifies mandatory requirements ("must-haves") for the first major release (1.0) of the Chassis/WWARN system for production use by WWARN.

The goal of the 1.0 release of Chassis/WWARN is to provide a system that supports the bare minimum needed to enable WWARN to begin accepting and curating data submissions. I.e., a system that is just sufficient to enable WWARN to establish key operating procedures and begin basic data management activities, and that can therefore be delivered as soon as possible. The functionality of the 1.0 release (as specified here) should therefore be limited to those features that are deemed absolutely necessary to enable the basic data management operations of WWARN to commence.

Subsequent releases of Chassis/WWARN will then provide additional functionality as needed. Specifying this additional functionality is not in scope for this document, however, some features that might be developed for future releases of Chassis/WWARN are briefly discussed.

The aim for the Chassis/WWARN 1.0 release is to coincide with the first WWARN milestone, which is scheduled for Q1 (Jan) 2010. However, because this document is in flux, and requirements may be added or removed, it is not possible to provide a precise release date at this stage. The informatics team will, however, provide access to working prototypes as often as possible on the way towards the 1.0 release, so the rest of the consortium can evaluate functionality and progress.

Note that, as this document currently stands, it is quite likely that the set of requirements is more than can be realistically be achieved by Q1 2010. Once the consortium has achieved some consensus on the key points of the data submission and curation workflows, it is suggested that we attempt to reduce and/or prioritise the functional requirements given below.

1.1.4. Limited Release for WHO Data Submission (0.1)

Note that a limited functionality release (0.1) of Chassis/WWARN, enabling submission of data using the WHO spreadsheet format only, is also planned. The requirements for Chassis/WWARN (0.1) will be specified in a separate document, however, they are expected to form a subset of the requirements specified here.

1.1.5. Managing User Accounts

This document assumes that management of user accounts, user roles, and authentication for the Chassis/WWARN system will be handled via the CGGH user account database and single sign-on system. This document therefore does not include any functionality for managing user accounts, assigning roles to users, or authenticating users (i.e., logging in and out of the system).

1.2. About This Document

1.2.1. Intended Audience

This document is an internal draft, intended for discussion within the WWARN consortium. It is meant to provide a concise description of the functionality provided by Chassis/WWARN 1.0. It is not intended for end-users of the Chassis/WWARN system.

1.2.2. Working Draft

This document is a working draft, and is intended to act as a focus for discussion within the WWARN consortium. Feedback is very welcome, and may be sent via email to informatics@wwarn.org.

Where there may be several possible choices relating to a particular requirement, some discussion is included in the document below, in the Discussion & Open Questions section.

1.2.3. Study Metadata (Study Site Questionnaire)

This document does not include a specification of the metadata fields that are to be included in the study site questionnaire for the Chassis/WWARN 1.0 release. It is suggested that a separate document be created to specify the SSQ for the Chassis/WWARN 1.0 release.

1.2.4. Chassis/*

This user requirements specification is scoped to the requirements of the WWARN data-sharing network. However, note that the Chassis/WWARN system is intended to be the first instance of a generic platform (Chassis/*), which can be customised to support the data management activities of other, similar, data-sharing networks. Therefore, this specification aims to describe requirements in a generic way, wherever possible, using terminology that is transferable to other data-sharing networks.

1.2.5. Structure of This Document

Section 2 (Workflow) summarises the key points of the workflow that the Chassis/WWARN system will be designed to support. It provides a summary of the vision for the Chassis/WWARN system.

Section 4 (Functional Requirements) breaks the workflow and use cases down into a set of functional requirements, which will be used by the development team to plan & prioritise.

Section 5 (Discussion) provides some information on discussion points and open issues.

2. Workflow - Summary of Key Points

This section summarises the key points of the data management workflow that Chassis/WWARN will be designed to support.

This section does not cover all requirements in detail. See the section below on use cases and functional requirements for detailed coverage of these points.

2.1. Roles & Responsibilities

The following roles are defined here for users of the Chassis/WWARN system.

  • A user is anyone who is authorized to access the Chassis/WWARN system.
  • A submitter is a user who can create a data submission. A submitter represents and is assumed to act for the people and/or organisations who have intellectual property rights and copyright over any data they submit.
  • A gatekeeper is a user with the authority to accept a data submission for curation. By accepting a submission, the gatekeeper is asserting that the submission is relevant to WWARN, and authorising the expenditure of resources (i.e., time and effort) on curating the submission.
  • A curator is a user who will perform curation activities, which might include (but are not limited to) reviewing submitted data, revising submitted data, ensuring that data meet quality control criteria and conform to standard data dictionaries.
  • A coordinator is a user who has responsibility for managing/coordinating some or all of the operations of a data-sharing network.
  • An administrator is a user with the authority to configure the Chassis/WWARN system.

Note that the same person may hold more than one of the above roles.

2.2. Data Submission Workflow

2.2.1. Creating Submissions

Data submission is the process whereby new data enters the Chassis/WWARN system.

A submitter initiates the process by creating a new submission. Creating a submission involves providing a title and textual summary for the submission, linking the submission to at least one study (see below), and linking the submission to at least one WWARN module. The submitter then uploads a data file containing the initial version of the submitted data.

2.2.2. Submission Ownership, Permissions and Privacy

When a new submission is created, the submitter who created it becomes the sole owner of that submission. If the submission owner wishes to work with another submitter to prepare the submission, the submission owner can grant ownership to other submitters. All owners will have the same permissions with respect to the submission as the initial owner, for example, they will all be able to edit the submission title and summary, and upload revisions to the submitted data (see below).

When a new submission is created, it is initially in a draft state. It is also private to its owner(s), and is not visible to any other users of the system.

2.2.3. Creating Studies

When a submitter is creating a new submission, they must link the new submission to at least one study.

Before creating the first submission for a study, where there is no previous record of the study in the Chassis/WWARN system, a submitter will have to create a record for the study. Creating a study involves providing at a title and textual summary for the study, and linking the new study to at least one WWARN module.

Once a study has been created, the submitter can then proceed to enter further metadata about the study (i.e., to fill out the study site questionnaire). However, to present the minimum obstacle to users, it is not necessary that any further study metadata be entered before a submitter can begin creating new submissions linked to the study, or request gatekeeper review for those submissions.

(Note that a gatekeeper could, on the other hand, require that some or all of the study site questionnaire be completed before a submission can be accepted for curation, i.e., completing some or all of the SSQ could form part of the gatekeeper's acceptance criteria, see below.)

2.2.4. Study Ownership, Permissions and Privacy

When a new study is created, the submitter who created it becomes the sole owner of that study. The study owner can also grant ownership to other submitters. All owners will have the same permissions with respect to the study as the initial owner, for example, they will be able to edit the study site questionnaire.

The owner(s) of a study can choose to make the study visible, in which case any user of the system will be able to find the study and view all details of the study site questionnaire, or discoverable, in which case any user of the system will be able to find the study and view its title and summary but will not be able to view any other information about the study, or private, in which case only the study owner(s) and authorised submitters will be able to find it and view its details.

If the study owners are aware that several submitters will be creating submissions linked to this study, then the owners can grant submission authority for that study to one or more other submitters.

However, in order to present the minimum obstacle to users, the system will allow any submitter to link their submissions to any visible or discoverable study, even if they are not an owner of that study, and even if they have not been granted submission authority by the study owner(s). In this case, the owner(s) of the study will be notified that the new submission has been linked to the study by an as-yet unauthorised submitter, and will be given the opportunity to grant submission authority to the submitter.

(Note also that a gatekeeper could require that the submitter has the appropriate authorisation from the study owner(s) beforing accepting a submission for curation, i.e., being an owner or authorized submitter for the study or studies to which a submission is linked could form part of the gatekeeper's acceptance criteria, see also below.)

2.2.5. Submitted Data

The first step after creating a submission is to upload a single data file containing the initial version of the submitted data.

Note that Chassis/WWARN (1.0) does not place any constraints on the nature or format of the data file uploaded as submitted data. The Chassis/WWARN (1.0) system will not attempt to parse the submitted data file, and will not provide any previews of the submitted data. Later versions of Chassis/WWARN may provide support for parsing and previewing of some data files, e.g., where data files conform to a standard data dictionary, or where data files contain a single table of data in a spreadsheet format.

2.2.6. Data Dictionaries

The Chassis/WWARN system requires that submitted data must be linked to a data dictionary, before a submission can be reviewed by a gatekeeper and accepted for curation (see also below).

Creating a data dictionary involves providing a title and textual summary for the dictionary, linking the dictionary to at least one WWARN module, and uploading a file containing the initial version of the dictionary.

A data dictionary defines the structure (i.e., layout, format) and semantics (i.e., meaning) of submitted data. It is expected that a data dictionary will at least provide a list of the field or variable names used in the data, and a textual definition of each field. Ideally, the dictionary would also specify the datatype of each field (e.g., integer, decimal, text, etc.), the units of the field (e.g., g/dl), and any constraints on the range of values for that field (e.g., numeric range constraints for a field with a numeric datatype, or controlled vocabulary constraints for a field with a text datatype).

Note, however, that Chassis/WWARN (1.0) does not place any restrictions on the nature or format of the data dictionary file uploaded when a new data dictionary is created or revised. The system will not attempt to parse the data dictionary file, nor provide any preview of its contents. Later versions of Chassis/WWARN may constrain the content and layout of data dictionary files, and parse files to store and provide views of data dictionaries. Later versions of the system may also provide online support for creating and editing the contents of data dictionaries, without needing to upload files. See also the discussion of data dictionaries below.

2.2.7. Data Dictionary Ownership, Privacy and Sharing

When a new dictionary is created, the submitter who created it becomes the sole owner of that dictionary. The dictionary owner can then grant ownership to other submitters. All owners will have the same permissions with respect to the dictionary as the initial owner, for example, the ability to revise the dictionary.

The owner(s) of a dictionary can choose to make the dictionary visible, in which case any user of the system will be able to find and view the dictionary, or discoverable, in which case any user of the system will be able to find the dictionary and view its title and summary but will not be able to view the content of the dictionary, or private, in which case only the dictionary owner(s) and authorised submitters will be able to find it and view its content.

A submitter can create a new data dictionary for each new submission, or re-use an existing data dictionary created by themselves, or re-use a dictionary created by someone else. In order to present the minimum obstacle to users, the system will allow any submitter to link their submitted data to any visible or discoverable dictionary. See also discussion of dictionaries and privacy below.

2.2.8. Gatekeeper Review

Before a submission can progress beyond the draft state and be accepted for curation, it must first be reviewed by a gatekeeper.

When a submission owner has finished preparing their submission, they can request a gatekeeper review of the submission. Until that point, the submission is in a draft state, and is private to the submission owner(s). Once the request has been made for gatekeeper review, a gatekeeper will be assigned to review the submission. The assigned gatekeeper may then view the details of the submission, the submitted data, the associated data dictionary, and the study(ies) to which the submission is linked.

The gatekeeper is responsible for performing an initial evaluation of the submitted data and accompanying data dictionary, to determine whether or not resources (i.e., time and effort) should be spent curating the data, and with what priority. The gatekeeper is not responsible for making any modifications to the data or approving the data for release to The WWARN Application.

The outcome of a gatekeeper review is either to accept the submission for curation, or not to accept the submission for curation. Once a gatekeeper has accepted a submission, they can assign a curator to the submission. Responsibility for the submission then passes from the gatekeeper to the assigned curator.

There may be more than one gatekeeper. For Chassis/WWARN, it is anticipated that each WWARN module will have its own gatekeeper(s), and that submissions will be assigned to a gatekeeper based on the module(s) to which the submission is linked by its submitter.

The criteria that the gatekeeper use to decide whether or not to accept a submission are referred to here as acceptance criteria. Each module may have different acceptance criteria. A module may also have no acceptance criteria, i.e., may accept all submissions by default.

A gatekeeper may also assign a priority to an accepted submission.

2.2.9. Revising Submissions

Before a submitter has requested gatekeeper review of a submission, they can upload revisions to the submitted data. A submitter can also make revisions after a submission is accepted for curation.

If a submission is not accepted, a submitter can revise a submission after the gatekeeper's review, and make a new request for gatekeeper review of the revised submission.

2.3. Data Curation Workflow

2.3.1. Curation Overview

The curation process takes submitted data as input and produces data that is suitable for release to The WWARN Application, and hence made available for visualisation, analysis, aggregation, sharing and publication.

The curation process comprises a wide and open-ended range of possible activities. However, in general, curation activities are either revision activities, which involve revising the submitted data in some way (e.g., adding, removing, or restructuring data), or review activities, where the submitted data are reviewed with respect to some criteria (e.g., ethical criteria, quality control criteria).

2.3.2. Revision Activities

Revision activities are curation activities which involve revising the submitted data in some way, e.g., adding, removing or restructuring the data, for any reason.

Note that the Chassis/WWARN 1.0 system will not provide any on-line support for revising submitted data. It is assumed that all revision activities will be achieved by downloading a chosen version of the submitted data to work on, making the necessary changes using other tools, e.g., off-line desktop tools, then uploading the revised version of the submitted data to the Chassis/WWARN system. See also the note on derivation and provenance below.

2.3.3. Transformation and Mapping to WWARN Standard Data Dictionaries

In addition to the non-standard data dictionaries that submitters can create, the Chassis/WWARN system has one or more data dictionaries managed by the curators that are agreed WWARN standard data dictionaries. WWARN standard data dictionaries are like any other data dictionaries, in that they define expectations for the structure and semantics of submitted data, except that they are maintained by the curators.

A special type of revision activity involves transforming data that is submitted in a non-standard form into something that conforms with one of the standard data dictionaries. Defining such a transformation is also referred to here as "mapping".

In the Chassis/WWARN 1.0 release, no support will be provided for mapping and transforming data. I.e., it is assumed that all data will be transformed using other systems, e.g., off-line desktop tools. Later versions of Chassis/WWARN may provide online support for defining and executing mappings between data conforming to different data dictionaries.

2.3.4. Review Activities

Review activities are curation activities where submitted data are reviewed with respect to some criteria, e.g., ethical or quality control criteria, and an assertion is made that the data either do or do not meet those criteria.

2.3.5. Validation

A special type of review activity involves checking that submitted data conform with respect to a given data dictionary, also referred to here as "validation".

Note that the Chassis/WWARN 1.0 release will not provide any support for validation. I.e., it is assumed that validation will be performed using other systems, e.g., by manual review, or using off-line desktop tools. Later versions of Chassis/WWARN may provide some online support for validation, where possible.

Note that, just because data conforms to the structural and semantic conditions of a standard data dictionary, does not necessarily mean that the data are reliable or make any sense. I.e., quality control is a separate concern from mapping submitted data to standard data dictionaries (see also below).

2.3.6. Release Criteria

A submission cannot be released to The WWARN Application unless the submitted data conforms to one of the WWARN standard data dictionaries. Therefore, the minimum curation activity for a submission is to check that the submitted data conforms to a WWARN standard data dictionary, and if not, to transform the data into something that does.

In addition to checking that submitted data conform to one of the WWARN standard data dictionaries, the curators may also define additional release criteria for submissions, which might include, e.g., ethical and/or quality control criteria. In order to make those criteria explicit, the curators may document them in a release criteria specification. The Chassis/WWARN system will provide support for creating and revising release criteria specifications.

The curators may then review any submission, with respect to a given release criteria specification, and assert whether or not the submitted data are approved with respect to those release criteria.

The curators may define more than one release criteria specification, e.g., to represent different degrees of confidence in the submitted data. The Chassis/WWARN system will expose information about which submissions have been approved by the curators with respect to which release criteria specification to The WWARN Application, which will enable users of The WWARN Application to filter data in visualisations and analyses by different release criteria.

Release criteria specifications will also be versioned, to enable criteria to be modified or updated without confusing the status of previously reviewed data.

Each WWARN module may also define and maintain its own release criteria specifications.

2.3.7. Making Studies and Submissions Public

All submissions are, by default, private, which means they are visible only to their owner(s), and to the Chassis/WWARN gatekeepers and curators. It is anticipated that submissions which are private, but where the submitted data conforms to a standard data dictionary, will be made available for analysis and visualisation by The WWARN Application (although possibly not the initial release of The WWARN Application), but that The WWARN Application will at all times respect the privacy constraints that are defined within the Chassis/WWARN system. I.e., in the case of private submissions, only the submission's owners will be able to view visualisations and analyses of data from their submission, and the data will not be included in any public aggregations.

A submission's owner(s) may choose to make their submission public at any time. Where a submission has been made public, and where the submitted data conforms to a standard data dictionary, it is assumed that the data will then be consumed and made publicly available by The WWARN Application, and included in public visualisations, aggregations and analyses.

A submission can only be made public if at least one of the studies to which the submission is linked has also been made public. If a study has been made public, then it is assumed that some details of the study will appear in the public areas of The WWARN Application.

2.3.8. Curation and Collaboration

See the discussion of curation and collaboration below.

2.3.9. Ordering of Curation Activities

The Chassis/WWARN system does not prescribe the order in which data are mapped to a standard data dictionary and revised to meet quality control standards and other release criteria. I.e., data may be submitted in a non-standard form, revised to meet quality control standards, and then mapped to a standard data dictionary; or data may be first mapped to a standard data dictionary, and then revised to meet quality control criteria; or revisions to address quality issues may be applied both before and after mapping to a standard data dictionary.

2.3.10. Provenance and Derivation of Data

A key design goal for Chassis/WWARN is that the complete and true derivation of any data released to The WWARN Application must be recorded. By derivation, we mean the complete sequence of events and revisions that led from the data as initially submitted to the data in its released form, i.e., its provenance.

As mentioned above, a curator can upload revisions of submitted data for any curation they are working on. To ensure that the true derivation of any data revision is captured, the curator must explicitly assert the derivation of any new revision they make, i.e., they must select the version of the data (either submitter version or another curator version) that was taken as input to the curator's new revision. This also ensures that curators can go back and do work on versions of the submitted data other than the latest version, and an accurate audit trail of their activities will still be maintained. (I.e., for any one submission, the curators' revisions to the submitted data are assumed to form an arbitrarily branching tree, where the root of each tree is one of the submitter's revisions.)

A submission owner can also upload a revision of the submitted data at any time. To keep things simple for a submitter, they do not have to assert the derivation of each new revision. (I.e., for any one submission, the submitter's revisions of the submitted data are assumed to form a single linear version series.)

3. Use Cases

The use cases below provide a more detailed illustration of how a user will interact with the Chassis/WWARN (1.0) system, in the most commonly anticipated situations. The use cases are not exhaustive, i.e., are not intended to cover all aspects of the system functionality.

3.1. Data Submission Use Cases

3.1.1. Use Case: Submitter Creates a New Study (1)

A submitter creates a new study, providing a title and textual summary for the study, and linking the new study to at least one module.

The submitter then proceeds to fill out the study site questionnaire.

3.1.2. Use Case: Submitter Creates a New Study (2)

A submitter creates a new study, providing a title and textual summary for the study, and linking the new study to at least one module.

The submitter does not immediately proceed to fill out the study site questionnaire. Rather, she begins creating submissions and linking them to the new study. At a later time, she comes back to the study to complete the study site questionnaire.

3.1.3. Use Case: Submitter Creates a New Submission With a New Data Dictionary

A submitter creates a new submission, provides a title for the submission, links the submission to at least one study and at least one module, and uploads a file containing the initial version of the submitted data.

In this case, no data dictionary is already present in the system that corresponds to the submitted data. The submitter creates a new data dictionary, provides a title for the dictionary, and uploads a file containing the initial version of the data dictionary. The submitter then links the initial version of the submitted data to the initial version of the new data dictionary.

The submitter then makes a request for gatekeeper review of the submission.

3.1.4. Use Case: Submitter Creates a New Submission With an Existing Data Dictionary

A submitter creates a new submission, provides a title for the submission, links the submission to at least one study and at least one module, and uploads a data file containing the initial version of the submitted data.

In this case, a data dictionary was previously created for a different submission that also applies to the new submission. The submitter links the initial version of the submitted data to the appropriate version of the existing data dictionary.

The submitter then makes a request for gatekeeper review of the submission.

3.1.5. Use Case: Submitter Revises a Submission Before Requesting Gatekeeper Review

A submitter creates a new submission, with a new data dictionary.

Before the submitter makes a request for gatekeeper review, she realises that some data are missing in the initial version of the submitted data. She downloads the initial version of the submitted data, adds the missing data, and uploads a revised version of the submitted data. She then links the revised version of the submitted data to the initial version of the data dictionary, because neither the structure nor the semantics of the data have changed.

The submitter then makes a request for gatekeeper review of the submission.

3.1.6. Use Case: Submitter Revises a Submission and Dictionary Before Requesting Gatekeeper Review

A submitter creates a new submission, with a new data dictionary.

Before the submitter makes a request for gatekeeper review, she realises that some fields are missing in the initial version of the submitted data. She downloads the initial version of the submitted data, adds the missing fields and corresponding data, and uploads a new version of the submitted data.

Because the structure of the data has changed, the submitter also uploads a new version of the data dictionary she previously created. The new version includes a definition of each of the new fields. She then links the revised version of the submitted data with the revised version of the data dictionary.

The submitter then makes a request for gatekeeper review of the submission.

3.1.7. Use Case: Gatekeeper Reviews and Accepts a Submission

A submitter makes a request for gatekeeper review of her new submission. The request contains a link to a specific version of the submitted data, and a specific version of the associated data dictionary, as specified in the request by the submitter.

A gatekeeper is allocated to review the request. She downloads the specified version of the submitted data and the specified version of the data dictionary, reviews them, and decides to accept the submission for curation. The gatekeeper then assigns a curator to the submission, and curation begins.

3.1.8. Use Case: Gatekeeper Accepts a Submission After Revision

A submitter makes a request for gatekeeper review of their new submission.

A gatekeeper is allocated to review the submission. She downloads the specified version of the submitted data and the specified version of the data dictionary, and reviews them.

In this case, the data dictionary is not complete, and the gatekeeper cannot understand the data. The gatekeeper decides not to accept the submission, explaining that the data dictionary is incomplete, and encouraging the submitter to revise the data dictionary and make a new request for gatekeeper review.

The submitter is notified of the outcome of the gatekeeper's review. She then uploads a new version of the data dictionary containing the missing information, removes the link from the initial version of the submitted data to the initial, incomplete, version of the data dictionary, and links the initial version of the submitted data to the revised version of the data dictionary.

The submitter then sends a new request for gatekeeper review of the revised submission.

A gatekeeper is notified of the request for review, downloads the initial version of the submitted data and the revised version of the data dictionary, reviews them, and decides to accept the submission. The gatekeeper assigns a curator, and curation begins.

3.1.9. Use Case: Submitted Data is Not Relevant to WWARN

A submitter makes a request for gatekeeper review of her new submission.

A gatekeeper is allocated to review the submission. She downloads the specified version of the submitted data and the specified version of the data dictionary, and reviews them.

In this case, it is clear that none of the data are relevant to any of WWARN's scientific modules. The gatekeeper decides that the submission will not be accepted, providing an explanation of the reasons why the submission was not accepted. The submitter is notified of the outcome of the gatekeeper's review and their explanation of the outcome. The submitter deletes her submission.

3.1.10. Use Case: Coordinator Overview of Submission Activity

A coordinator looks through a list of all submissions that are pending gatekeeper review. She notices that one submission has been pending gatekeeper review for more than 2 weeks. She contacts the gatekeeper assigned to review the submission and asks that the submission be reviewed urgently.

3.1.11. Use Case: Gatekeeper Overview of Submission Activity

A gatekeeper looks through a list of all submissions pending gatekeeper review that have been allocated to her and that have not yet been reviewed. She selects a submission from the list and reviews it.

3.1.12. Use Case: Submitter Overview of Submission Activity

A submitter looks through a list of all her submissions that have been reviewed by a gatekeeper and accepted for curation, a list of submissions that have been reviewed and not accepted, a list of all her submissions that are pending gatekeeper review, and all her draft submissions (for which she has not yet requested gatekeeper review).

From the list of her draft submissions, she selects those that are ready for gatekeeper review, and requests gatekeeper review for them.

3.2. Data Curation Use Cases

Note that some details of the use cases below are simplified, to avoid distracting from the key features of the use case that are relevant to the design of Chassis/WWARN. For example, the reasons why a curator might need to make revisions to submitted data, and the nature of the revisions they might need to make, will be far more varied and subtle than are suggested below.

3.2.1. Notation Conventions

In the use cases below, letters A, B, C, etc., are used to help identify different revisions of submitted data, e.g., "the initial revision of the submitted data (A) was revised by the curator to create a new revision (B)". If the letter has the superscript "s" (e.g., As), then the revision is linked to a standard data dictionary. If no superscript is present, the data is linked to a non-standard data dictionary.

3.2.2. Use Case: Curator Manages a Standard Data Dictionary

A curator creates a new standard data dictionary, provides a title and textual summary for the dictionary, links the dictionary to one or more modules, and uploads a file containing the initial version of the dictionary.

At a later time, the WWARN curators agree to make an update to this standard data dictionary, to incorporate some new fields as standard. The curator uploads a new file containing the revised version of the data dictionary, and enters a log message describing the changes.

3.2.3. Use Case: Curator Manages a Release Criteria Specification

A curator creates a new release criteria specification, provides a title and textual summary for the specification, links the specification to the module to which it applies, and uploads a file containing the initial version of the specification.

At a later time, the curators agree to make the release criteria more stringent, including new quality control criteria. The curator uploads a new file containing the revised version of the release criteria specification, and enters a log message describing the changes.

The curators review all prior data submissions that were approved with respect to the old version of the release criteria, to determine if they also can be approved with respect to the updated version of the release criteria.

3.2.4. Use Case: Standard Submission, Requires No Curation

A submitter requests gatekeeper review for a new submission, where the submitted data is already linked by the submitter to a WWARN standard data dictionary. A gatekeeper accepts the submission and assigns a curator.

The assigned curator reviews the initial version of the submitted data (As), and finds that no changes need to be made. She reviews the submitted data with respect to the latest version of the appropriate release criteria specification, and asserts that the release criteria have been met.

The submitter sees that her initial version of the submitted data (As) has been approved for release to The WWARN Application, and that no other actions were taken.

3.2.5. Use Case: Curation of Standard Submission

A submitter requests gatekeeper review for a new submission, where the submitted data is already linked to a WWARN standard data dictionary. A gatekeeper accepts the submission and assigns a curator.

The assigned curator downloads the initial version of the submitted data (As), reviews it, and finds that some rows have missing data values. She removes the rows with missing data, uploads a new data file containing the revised data (Bs), links it to the version of the submitted data from which it was derived (As), and links it to the standard data dictionary. She also enters a log message explaining what changes were made and why.

The assigned curator then reviews the submitted data (Bs) with respect to the latest version of the appropriate data release criteria, and asserts that the release criteria have been met.

The submitter sees that her submission has been approved for release to The WWARN Application, after the assigned curator made a revision (Bs) to her initial version (As). She downloads the curator's revision (Bs) to inspect the changes.

3.2.6. Use Case: Standard Submission Requires Submitter Revision

A submitter requests gatekeeper review for a new submission, where the submitted data is already linked to a WWARN standard data dictionary. A gatekeeper accepts the submission and assigns a curator.

The assigned curator downloads the initial version of the submitted data (As), reviews it, and finds that some rows have data values that appear to be out of range. She decides to contact the submitter, and notify her of the problem.

The submitter realises that some errors were made in the initial version of the submitted data (As). The submitter downloads her initial version of the submitted data (As), fixes the errors, and uploads a new version of the submitted data (Bs).

The assigned curator then reviews the submitted data (Bs) with respect to the latest version of the appropriate data release criteria, and asserts that the release criteria have been met.

The submitter sees that her submission has been approved for release to The WWARN Application, after she made a revision (Bs) to her initial version (As).

3.2.7. Use Case: Curation of Non-Standard Submission

A submitter requests gatekeeper review for a new submission with a new data dictionary. The data dictionary is not one of the WWARN standard data dictionaries. A gatekeeper accepts the submission and assigns a curator.

The assigned curator downloads the initial version of the submitted data (A) and the initial version of the data dictionary. She finds that several rows appear to be missing some data values. She removes the rows with missing data, uploads her revised version of the submitted data (B), links it to the version of the submitted data from which it was derived (A), links it to the non-standard data dictionary, and enters a log message describing the changes she has made and the reasons for making the changes.

The assigned curator then decides to map the submitted data onto the appropriate standard data dictionary. She downloads her curated revision of the submitted data (B), and downloads the latest version of the appropriate standard data dictionary. Using her favourite desktop data transformation tool, she creates a new data file conforming to the standard data dictionary. She then uploads this new data file (Cs), links it to the version of the submitted data from which it was derived (B), links it to the latest version of the standard data dictionary, and enters a log message describing the transformation she performed.

The assigned curator now believes the data is ready for release to The WWARN Application. She reviews her standardised version of the submitted data (Cs) with respect to the latest version of the appropriate release criteria specification, and asserts that the release criteria have been met. The data are then made available for export to The WWARN Application.

The submitter sees that her initial version of the submitted data (A) has undergone one curation revision (B, derived from A), one standardisation revision (Cs, derived from B), and has been approved for release to The WWARN Application.

3.2.8. Use Case: Non-Standard Submission, Curation and Repeated Standardisation

A submitter requests gatekeeper review for a new submission with a new data dictionary. The data dictionary is not one of the WWARN standard data dictionaries. A gatekeeper accepts the submission and assigns a curator.

The assigned curator downloads the initial version of the submitted data (A) and makes a revision (B), without changing the structure of the data.

The assigned curator then decides to map the submitted data onto the appropriate standard data dictionary. She downloads her curated revision of the submitted data (B), and creates a new data file conforming to the appropriate standard data dictionary. She then uploads this new data file (Cs), links it to the version of the submitted data from which it was derived (B), links it to the latest version of the standard data dictionary, and enters a log message describing the transformation she performed.

The assigned curator now believes the data is ready for release to The WWARN Application. She reviews her standardised version of the submitted data (Cs) with respect to the latest version of the appropriate release criteria specification, but finds some problems in the standardised data. She realises that the problems are due to the standardisation transformation she applied. She goes back to the curated, non-standard revision of the submitted data (B), and repeats her transformation, correcting the previous errors. She uploads this new standardised revision of the submitted data (Ds), links it to the version of the submitted data from which it was derived (B), links it to the latest version of the standard data dictionary, and enters a log message describing the transformation she performed.

The assigned curator reviews her corrected standardised revision of the submitted data (Ds) with respect to the latest version of the appropriate release criteria specification, and asserts that the release criteria have been met.

The submitter sees that her initial version of the submitted data (A) has undergone one curation revision (B, derived from A), one standardisation revision (Cs, derived from B), a second standardisation revision (Ds, derived from B), and has been approved for release to The WWARN Application.

3.2.9. Use Case: Non-Standard Submission, Curation After Standardisation (1)

A submitter requests gatekeeper review for a new submission with a new data dictionary. The data dictionary is not one of the WWARN standard data dictionaries. A gatekeeper accepts the submission and assigns a curator.

The assigned curator downloads the initial version of the submitted data (A), reviews it, and decides that there are some problems, but that they will be easier to deal with after the submission has been standardised. She creates a new data file (Bs) conforming to the appropriate standard data dictionary, uploads it, links it to the version of the submitted data from which it was derived (A), links it to the latest version of the standard data dictionary, and enters a log message describing the transformation she performed.

The assigned curator then creates a new revision (Cs), derived from her standardised revision (Bs), fixing the problems she identified earlier. She approves this new version (Cs) for release.

The submitter sees that her initial version of the submitted data (A) has undergone one standardisation revision (Bs, derived from A), one curation revision (Cs, derived from Bs), and has been approved for release to The WWARN Application.

3.2.10. Use Case: Non-Standard Submission, Curation After Standardisation (2)

A submitter requests gatekeeper review for a new submission with a new data dictionary. The data dictionary is not one of the WWARN standard data dictionaries. A gatekeeper accepts the submission and assigns a curator.

The assigned curator downloads the initial version of the submitted data (A), reviews it, and decides to map the file onto the appropriate standard data dictionary. She creates a new data file (Bs), uploads it, links it to the version of the submitted data from which it was derived (A), links it to the latest version of the standard data dictionary, and enters a log message describing the transformation she performed.

The assigned curator then reviews her standardised revision (Bs) and finds some problems. She decides that the problems are best fixed by going back to the initial, non-standard version of the submitted data (A). She creates a new, non-standard revision of the submitted data (C), uploads it, links it to the version of the data from which it was derived (A), links it to the appropriate version of the non-standard data dictionary, and enters a log message.

The assigned curator then maps the new non-standard revision of the submitted data (C) onto the appropriate standard data dictionary. She creates a new data file (Ds), uploads it, links it to the version of the submitted data from which it was derived (C), links it to the latest version of the standard data dictionary, and enters a log message describing the transformation she performed.

The assigned curator then approves her latest version of the standardised data (Ds) for release.

The submitter sees that her initial version of the submitted data (A) has undergone one standardisation revision (Bs, derived from A), one curation revision (C, derived from A), a second standardisation revision (Ds, derived from C), and has been approved for release to The WWARN Application.

3.2.11. Use Case: Curator Delegation

A submitter requests gatekeeper review for a new submission with a new data dictionary. The data dictionary is not one of the WWARN standard data dictionaries. A gatekeeper accepts the submission and assigns a curator.

The assigned curator delegates the initial review of the submitted data to another curator who specialises in quality aspects of that type of data. The delegate curator downloads the initial version of the submitted data (A), reviews it, decides that some rows do not meet quality control criteria, removes them, and uploads a revised version of the submitted data (B), notifying the assigned curator that their task is complete.

The assigned curator then delegates the mapping of the data onto a standard data dictionary to a different curator, who specialises in transforming data and data standards. The delegate curator downloads the revised version of the submitted data (B), transforms them to conform to the latest version of the appropriate standard data dictionary, and uploads the transformed data (Cs), notifying the assigned curator that their task is complete.

The assigned curator then delegates the final review of the submission with respect to the appropriate release criteria to third delegate curator. The delegate approves the standardised version of the submitted data (Cs) for release.

(See also the discussion of curation and collaboration below.)

3.2.12. Use Case: Coordinator Oversight of Curation and Standardisation Activities

A coordinator reviews a list of all submissions for a given module that have been accepted by a gatekeeper, but have not yet been approved for release. She sees which of these have data linked to a standard data dictionary, and which do not. From those that do not, she selects one from the list to investigate in more detail.

For the selected submission, the coordinator reviews a log of all curation and standardisation activities for that submission. She sees that several curation revisions have been made to the submitted data, but that no standardised revisions have yet been made. She contacts the assigned curator to ask for more information.

3.2.13. Use Case: Curator Oversight of Curation & Standardisation Activities

A curator views a list of all submissions for which she is the assigned curator, and that have not yet been approved for release. From these, she then views those that have not yet been mapped onto a standard data dictionary. She selects a submission from this list, and begins working on it.

4. Functional Requirements

This section lists mandatory functional requirements for the Chassis/WWARN system, based on the workflow and the use cases described above.

4.1. Data Submission Functional Requirements

4.1.1. (Submitter) Managing Studies

  • A submitter will be able to create a new study. This involves providing a title and textual summary for the study, linking the study to at least one module, and specifying whether the study is visible, discoverable or private.
  • A submitter will be able to grant ownership to any other submitter for any study they own.
  • For any study they own, a submitter will be able to authorise any other submitter to link submissions to that study.
  • A submitter will be able to view a list of studies they own.
  • A submitter will be able to view a list of studies created by anyone, including all visible and discoverable studies, and all private studies which they own or are authorised to see.
  • Lists of studies can be filtered by module.

4.1.2. (Submitter) Entering and Revising Study Metadata

  • A submitter will be able to enter metadata (i.e., fill out the SSQ) for any study they own.
  • A submitter will be able to revise (i.e., update, modify) metadata for any study they own.

See also the note on study metadata above.

4.1.3. (Submitter) Managing Submissions

  • A submitter will be able to create a new submission. This involves providing a title and textual summary for the submission, associating the submission with at least one previously created study, associating the submission with at least one WWARN module, and uploading a single file containing the initial version of the submitted data.
  • A submitter will be able to grant ownership to any other submitter for any submission they own.
  • A submitter will be able to view a list of all submissions they own.

4.1.4. (Submitter) Revising Submitted Data

  • A submitter will be able to upload a data file containing a new version (i.e., revision) of the submitted data, for any submission they own.
  • A submitter will be able to view a version history of the submitted data for any submission they own.

4.1.5. (Submitter) Downloading Submitted Data

  • A submitter will be able to download the latest version of the submitted data, for any submission they own.
  • A submitter will also be able to download any previous version of the submitted data, for any submission they own.

4.1.6. (Submitter) Managing Data Dictionaries

  • A submitter will be able to create a new data dictionary. This involves providing a title and textual summary for the new data dictionary, uploading a file containing the initial version of the data dictionary, and specifying whether the dictionary is visible, discoverable, or private.
  • A submitter will be able to grant ownership to any other submitter for any dictionary they own.
  • A submitter will be able to grant visibility to any other submitter for any private or discoverable dictionary they own.
  • A submitter will be able to view a list of all data dictionaries they own.
  • A submitter will be able to view a list of all data dictionaries created by anyone, including all visible and discoverable dictionaries, and all private dictionaries they own or are authorised to see.

4.1.7. (Submitter) Revising a Data Dictionary

  • A submitter will be able to upload a file containing a new version of a previously created data dictionary they own.
  • A submitter will be able to view a version history for any data dictionary they own.

4.1.8. (Submitter) Downloading a Data Dictionary

  • A submitter will be able to download any version of any visible data dictionary, and any private or discoverable dictionary they are authorised to see.

4.1.9. (Submitter) Linking Submitted Data With a Data Dictionary

  • For any submission they own, a submitter will be able to link a specific version of the submitted data with a specific version of a data dictionary. They do not have to also own the data dictionary to which the link is made.
  • For any submission they own, a submitter will be able to remove the current link between the latest version of the submitted data and the version of the data dictionary to which it is linked, and replace it with a new link to a different version of the same data dictionary, or some version of a different data dictionary.
  • For any submission they own, a submitter will be able to view a history of any changes that have been made to links between the submitted data and data dictionaries.

4.1.10. (Submitter) Requesting Gatekeeper Review

  • A submitter will be able to request gatekeeper review for any submission they own.

4.1.11. (Submitter) Overview of Submission Activity

  • A submitter will be able to view a list of all submissions they own. The status of each submission, i.e., either draft (not yet submitted for gatekeeper review), pending gatekeeper review, or gatekeeper reviewed, will be clearly shown. If the submission has been reviewed by a gatekeeper, the outcome of each review (either accepted or not accepted) will also be clearly shown.
  • A submitter will also be able to view a list of only draft submissions, a list of submissions that are pending gatekeeper review, a list of submissions that have been reviewed by a gatekeeper and accepted, and a list of submissions that have been reviewed by a gatekeeper and not accepted.
  • Any of these submission lists can be grouped by study or module.

4.1.12. (Gatekeeper) Overview of Submission Activity

  • A gatekeeper will be able to view a list of all submissions from any submitter, not including draft submissions. The status of each submission (i.e., either pending gatekeeper review, reviewed and accepted, or reviewed and not accepted) will be clearly shown.
  • A gatekeeper will also be able to view a list of only those submissions that have been reviewed and accepted (not necessarily by themselves), a list of submissions that have been reviewed and not accepted, and a list of submissions that are pending gatekeeper review.
  • Any of these submissions lists can be grouped by submitter, study or module.

4.1.13. (Gatekeeper) Downloading Submitted Data and Data Dictionaries

  • A gatekeeper will be able to download any version of submitted data for which a request for gatekeeper review has been made. A gatekeeper will also be able to download any version of a data dictionary to which the submitted data is linked.

4.1.14. (Gatekeeper) Reviewing Submissions

  • A gatekeeper will be able review any any submission for which a request for gatekeeper review has been made, and indicate that the outcome of the review is a decision to accept the submission, or a decision not to accept the submission.
  • A gatekeeper will be able to assign a curator to an accepted submission.
  • If the outcome of a gatekeeper review is a decision not to accept a submission, the gatekeeper will be able to provide an explanation of the reasons why the submission has not been accepted, and any further information for the submitter.

4.1.15. (Coordinator) Overview of Submission Activity

  • A coordinator will be able to view a list of all submissions from any submitter, not including draft submissions. The status of each submission (i.e., either pending gatekeeper review, or reviewed by a gatekeeper) will be clearly shown. Where the submission has been reviewed by a gatekeeper, the outcome of each review (either accepted or not accepted) will also be clearly shown.
  • A coordinator will also be able to view a list of only those submissions that have been reviewed by a gatekeeper and accepted, a list of submissions reviewed by a gatekeeper and not accepted, and a list of submissions that are pending gatekeeper review.
  • Any of these submissions lists can be grouped by submitter, study or module.

4.2. Data Curation Functional Requirements

4.2.1. (Curator) Creating a Standard Data Dictionary

  • A curator will be able to create a new standard data dictionary. This involves providing a title and textual summary for the new standard data dictionary, linking the dictionary to at least one module, and uploading a file containing the initial version of the data dictionary.

4.2.2. (Curator) Revising a Standard Data Dictionary

  • A curator will be able to upload a file containing a revised version of a standard data dictionary.
  • A curator will be able to view a version history for any standard data dictionary.

4.2.3. (Curator) Downloading a Standard Data Dictionary

  • A curator will be able to download any revision of a standard data dictionary.

4.2.4. (Curator) Creating a Release Criteria Specification

  • A curator will be able to create a new release criteria specification. This involves providing a title and textual summary for the new specification, linking the specification to exactly one module, and uploading a file containing the initial version of the specification.

4.2.5. (Curator) Revising a Release Criteria Specification

  • A curator will be able to upload a file containing a revised version of a release criteria specification.
  • A curator will be able to view a version history of any release criteria specification.

4.2.6. (Curator) Downloading a Release Criteria Specification

  • A curator will be able to download any revision of a release criteria specification.

4.2.7. (Curator) Downloading Submitted Data

  • A curator will be able to download any version of the submitted data for any submission they are assigned to.

4.2.8. (Curator) Revising Submitted Data

  • A curator will be able to upload a data file containing a new version of the submitted data, for any submission to which they have been assigned to. The curator must explicitly select which previous version of the submitted data they started work on, i.e., which previous version their new revision is most directly derived from. The curator will also be able to enter a log message describing the changes made.

4.2.9. (Curator) Linking Submitted Data With a Data Dictionary

  • A curator will be able to link a specific version of the submitted data to a specific version of a data dictionary (either standard or non-standard), for any submission to which they have been assigned or delegated to work.
  • A curator will be able to change the data dictionary link for any submission they are assigned to or delegated to work on.
  • A curator will be able to view a history of changes made to the data dictionary link, for any submission they are assigned to or delegated to work on.

4.2.10. (Curator) Viewing Submitted Data Version History

  • A curator will be able to view a complete derivation tree of the submitted data, for any submission to which they have been assigned or delegated to work on. It will be clear which versions are linked to a standard data dictionary, and which are not.

4.2.11. (Curator) Reviewing Data for Release

  • A curator will be able to review submitted data w.r.t. some version of a release criteria specification, and assert that the data either do or do not meet the specified criteria, for any submission to which they have been assigned.

4.2.12. (Curator) Delegation

  • An assigned curator will be able to delegate tasks to other curators, for any submission to which they have been assigned.
  • A curator will be able to indicate completion of a delegated tasks.

See also the discussion of curation and collaboration below.

4.2.13. (Curator) Overview of Curation Activity

  • A curator will be able to view a list of all submissions that have been accepted but not yet approved for release, and a list of those that have been approved for release, for any version of release criteria, for any submission to which they are assigned. Those for which a revision is available conforming to a standard data dictionary will be clearly marked.
  • These lists can be grouped by module, study or submitter.

4.2.14. (Coordinator) Overview of Curation Activity

  • A coordinator will be able to view a list of all submissions that have been accepted but not yet approved for release, and a list of those that have been approved for release, for any version of release criteria. Those submissions for which a revision of the submitted data is available conforming to a standard data dictionary will be clearly marked.
  • These lists can be grouped by module, study or submitter.

5. Discussion & Open Questions

5.1. Constraints on Data Dictionaries

This document tentatively takes the position that, for the first major release of Chassis/WWARN, there will be no constraints on the structure or content of data dictionaries. Data dictionaries are simply uploaded as files of any type, with any content. This position was taken because the complete range of possible submitted data structures is not currently known, and therefore it may be useful in the early stages of WWARN to enable some flexibility as the consortium encounters new data and works towards a common approach to data dictionaries.

However, the goal for later releases of Chassis/WWARN is to support a single type of data dictionary, with the content of the data dictionary managed through a web interface, rather than through the upload of files. Based on previous experience in CGGH, the likely approach would be that each such data dictionary would define a set of variables (also known as fields or attributes), where each variable has a unique identifier, a human-readable label, a human-readable definition of the variable's meaning, a datatype for the variable, the units of the variable, and any range constraints (e.g., numeric constraints or controlled vocabulary) on the values the variable may take. This would enable further functionality, such as an analysis of the diversity of variable names used across all data submitted, which can support the process of convergence on a common set of data standards across the consortium. It would also be possible in some cases to use the data dictionary to drive automated functionality such as simple validation of datatypes and data ranges. It would also provide a basis for developing functionality to define and execute mappings (i.e., transformations) between data conforming to different data dictionaries, which is planned for future releases of Chassis/WWARN.

Placing no constraints on the content or structure of data dictionaries, in the first major release of Chassis/WWARN, may however have a number of undesirable consequences. Firstly, many submitters may not have prior experience of constructing data dictionaries, and may not know what information to provide or how to provide it. Without clear guidance and examples, both the quality (usefulness), the structure and the content of data dictionaries created by submitters may be highly variable. Whilst prominent guidance and examples may ameliorate some of these issues, there is no guarantee on consistency or convergence towards a common approach to constructing data dictionaries. Secondly, without any constraints on data dictionaries, it will not be possible to reliably parse or analyse all data dictionaries, or to use the data dictionary to support any automated functionality such as simple validation of datatypes and data ranges. Thirdly, once a common approach to data dictionaries is agreed and supported in Chassis/WWARN, there would be a legacy of previously created dictionaries in a variety of formats to migrate, which would require some effort.

Whether or not the first major release of Chassis/WWARN enforces a single approach to data dictionaries, or is more flexible and places no constraints on files uploaded as data dictionaries, is therefore an important open question in this specification.

5.2. Curation and Collaboration

A design goal for the Chassis/WWARN system is that, for any submission under curation, it is always clear who has responsibility for progressing that submission forward. Thus, when the gatekeeper decides to accept a submission, she also assigns a curator to the submission, and responsibility passes to the assigned curator. The assigned curator then clearly sees which submissions she has to work on.

If the assigned curator needs to work with other curators to complete the work on a submission, there are at least two models under which they could operate.

Under the sequential model, the assigned curator passes responsibility to another curator, who completes some work on the submission, then either passes responsibility back to the original curator or on to yet another curator. Only one curator at any one time has ownership of the submission and can upload revisions of the submitted data.

Under the parallel model, the assigned curator retains overall responsibility for the submission, but grants permission to one or more other curators to carry out work on the submission in parallel.

This begs the following questions. Which curators can view which submissions? Which curators can upload revisions to which submissions? How do we track who has responsibility for a submission at any given time, and how do they pass responsibility to others?

5.3. Data Dictionaries and Privacy

An assumption above is that some submitters will not want to share the content of their data dictionaries with other submitters. Therefore, submitters can make their dictionaries private. However, to balance the need for privacy with the need to present as few obstacles as possible to the user, the document above introduces the ability to make a dictionary discoverable, which means that other users can find it, but cannot see its content. Does it make sense for some dictionaries to be discoverable? Or should we just stick with dictionaries being either visible or private? If we stick with the notion of discoverable dictionaries, what happens if a submitter links their submitted data to a discoverable data dictionary? Does the dictionary owner receive a notification, or request to grant access to the dictionary?
